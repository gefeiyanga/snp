# 房贷/车贷标准摊销与动态剩余本金 — 设计说明

**日期**：2026-04-22  
**状态**：已定稿（经需求澄清）  
**范围**：SNP 应用内「负债」模块中与房贷、车贷相关的数据模型、计算口径与交互；不含提前还款、利率重定价等非 v1 能力。

---

## 1. 背景与目标

### 1.1 问题

用户录入车贷/房贷时，合同本金往往对应**过去某一时刻**的放款额；至「当前」已有若干期月供偿还本金。现有实现中 `remaining` 主要依赖手工维护，**不会**随时间自动反映已偿还期数。

### 1.2 目标

- 对 **房贷**、**车贷** 两类负债，在录入完整摊销参数后，**剩余本金**随日历自动演进。
- **信用卡**、**其他** 类别不参与摊销引擎，保持现有「手工剩余」逻辑。
- 计算采用 **标准摊销**（非拍脑袋线性扣减）。

### 1.3 非目标（v1 明确不做）

- 提前还款、缩期、部分还本、宽限期、罚息。
- 多段利率、公积金+商贷组合、气球贷等特殊品种。
- 与银行系统对账、自动拉取流水。

---

## 2. 需求结论（已确认）

| 项 | 结论 |
|----|------|
| 计算模型 | **B**：标准摊销（等额本息 / 等额本金公式）。 |
| 适用类别 | **房贷** 与 **车贷** 使用同一套引擎与字段规则。 |
| 还款方式 | **C**：每笔在录入时 **二选一** — `等额本息` 或 `等额本金`。 |
| 日期采集 | **A**：仅采集 **第一期还款日**；**起息日默认与第一期还款日为同一自然日**（v1 不单独录入起息日）。 |
| 「月末动态」 | 采用 **方案 1**：**以用户本地日历的「今天」为 asOf**；所有 **约定应还日 ≤ asOf** 的期数视为已偿还，并据此计算当前剩余本金。说明：这与口语「变化发生在月末」在字面上可能不完全一致；实际效果是 **按对日还款** 在任意一天打开应用即可看到与「已过应还日」一致的余额。若未来需要「关账日=自然月末」的展示，可作为独立可选策略另做开关（非 v1）。 |

## 3. 数据模型

在现有 `LiabilityRecord`（见 `src/types/ledger.ts`）基础上，对 **房贷/车贷** 摊销实例要求具备或可推导的字段如下（实现时可与已有字段合并命名，避免重复）：

| 字段 | 类型 | 说明 |
|------|------|------|
| `amount` | number | **合同初始本金**（与现有语义一致，可视为 `originalPrincipal`）。 |
| `remaining` | number | **当前剩余应还本金**；摊销类为 **派生值**（见 §4），由引擎根据参数与 asOf 计算；持久化策略见 §5。 |
| `firstPaymentDate` | string (ISO 日期) | **第一期还款日**；起息日与之相同，不单独存储。可与现有 `dueDate` 对齐或新增专用字段（实现时二选一并在代码中统一）。 |
| `termMonths` | number | 总还款期数（月），正整数。 |
| `interestRate` | number | **年名义利率（%）**；月利率 = `interestRate / 100 / 12`。 |
| `repaymentMethod` | enum | `equal_payment`（等额本息）\| `equal_principal`（等额本金）。 |
| `category` | string | 必须为 `房贷` 或 `车贷` 之一才启用摊销引擎。 |
| `monthlyPayment` | number | v1 建议：**只读展示**（由首期后第一期月供公式算出）；**不作为**摊销反推的输入源，避免与公式冲突。 |
| `legacyManual`（或等价标记） | boolean 可选 | 旧数据缺摊销字段时置 true，直至用户补全参数。 |

**信用卡 / 其他**：不读取 `repaymentMethod` / `termMonths` 等；`remaining` 仍以用户编辑为准。

---

## 4. 计算口径

### 4.1 月利率与期数

- `monthlyRate = annualInterestRate / 100 / 12`。
- 期数索引：`period = 1 … termMonths`，第 `period` 期的 **应还日** 由 `firstPaymentDate` **按月顺延**得到。

### 4.2 对月还款日（无同日时的规则）

若目标月份不存在与首期相同的「日」（如首期 1 月 31 日 → 2 月），**该期应还日取该自然月的最后一天**（v1 在 spec 层写死，避免实现分歧）。

### 4.3 asOf 与「已还期数」

- `asOf` = 用户设备本地时区的 **当前日期**（日期粒度即可，不含「必须 23:59」的额外要求，除非产品后续要求）。
- 设第 `k` 期的应还日为 `due(k)`。则 **已偿还的最大期数** `paidPeriods = max { k | due(k) ≤ asOf }`（若无满足条件的 k，则为 0）。
- **剩余本金** `remaining(asOf) = balanceAfterPeriod(paidPeriods)`，其中 `balanceAfterPeriod` 由所选 **等额本息** 或 **等额本金** 的标准递推/闭式公式给出（实现阶段给出具体公式与引用）。

### 4.4 等额本息 / 等额本金

- **等额本息**：用标准月供（PMT）公式；剩余本金可用年金现值递推或标准闭式。
- **等额本金**：每期偿还本金 = `amount / termMonths`；每期利息 = 当期期初剩余 × `monthlyRate`；逐期递推。

### 4.5 边界

- `termMonths`、`interestRate`、`firstPaymentDate`、`amount` 非法时禁止保存并提示。
- `paidPeriods ≥ termMonths` 时剩余本金为 0（允许极小浮点误差，展示时四舍五入到分）。

---

## 5. 持久化策略（推荐）

- **读取路径**（列表、总览、图表）：对房贷/车贷且非 legacy 记录，**始终**用 §4 根据当日 asOf **重算** `remaining`（及展示用月供），保证与参数一致。
- **写入路径**：保存/编辑参数后，将重算后的 `remaining`（及可选 `monthlyPayment` 展示缓存）**写回存储**，便于离线与其他未改摊销引擎的代码路径兼容。

若采用「仅计算不写回」，需在文档与代码中统一所有读路径，避免遗漏。

---

## 6. 用户界面与流程

- 当 `category` 为 **房贷** 或 **车贷** 时，表单展示：**还款方式（二选一）**、**合同本金**、**年利率**、**总期数（月）**、**第一期还款日**；**月供只读**（计算展示）。
- 当类别为 **信用卡 / 其他** 时，不展示摊销专用字段；`remaining` 可编辑，逻辑与现网一致。
- **编辑**摊销参数后：从首期起整表逻辑重算；若结果与用户心理预期差异大，以 Toast 或说明文案提示「按标准公式重算」。

---

## 7. 旧数据迁移

- 已存在、缺少 `termMonths` / `repaymentMethod` / 首期日 中任一项的房贷/车贷：标记为 **legacy**，**不**自动摊销，`remaining` 维持用户原值直至补全。
- 用户补全后取消 legacy，下一读即按 §4 计算。

---

## 8. 测试要点（实现阶段验收）

- 与表格软件或公开计算器对齐：固定本金、年利率、期数、首期日，等额本息 / 等额本金各至少 2 组，核对第 1、中间、最后一期后的剩余本金与月供。
- **对月无同日**（如 1/31 → 2 月）单测应还日与剩余本金。
- **asOf** 落在两期之间、恰为应还日、贷款已结清三种情况。
- legacy 补全字段后行为切换。

---

## 9. 已知限制（v1）

- 不考虑提前还款导致的重算；用户若提前大额还款，需 v2 或暂时用手工调整类别（不推荐混用，可在 UI 提示）。
- 方案 1 下，**不会在「物理上」等到月末才更新**；更新发生在 **每次按 asOf=今天重算** 时（通常即打开应用或进入负债页时）。
