# 维护型账本底座设计

**日期**：2026-05-08  
**状态**：第一阶段设计稿  
**范围**：为每日收支、每月工资、房贷/车贷还款等高频维护动作建立账户、流水、周期规则与待确认发生项。现有 `assets` / `liabilities` 继续保留，避免破坏用户本地数据。

## 1. 背景

现有应用以资产和负债快照为中心，适合查看总览，但不适合长期维护。真实使用中，用户每天发生收入/支出，每月固定发薪和还贷。如果每次都手动编辑资产或负债金额，维护成本会越来越高。

## 2. 产品原则

- 用户记录的是「事件」，不是反复维护结果。
- 资产、负债、净资产尽量从账户余额、流水、行情和贷款计划中推导。
- 周期事项先进入待确认队列，不静默写入真实账本。
- 新能力不改旧存储 key，旧快照数据继续可读可编辑。

## 3. 核心模型

### 3.1 AccountRecord

账户是余额载体，覆盖现金、银行卡、支付账户、信用卡、贷款、投资账户等。

关键字段：

- `type`：账户类型。
- `side`：`asset` 或 `liability`，决定余额含义。
- `openingBalance`：启用流水系统时的初始余额。
- `linkedAssetId` / `linkedLiabilityId`：后续与现有资产/负债快照打通。

### 3.2 TransactionRecord

流水是维护型账本的事实来源。

支持类型：

- `income`：收入，例如工资。
- `expense`：支出，例如餐饮、交通。
- `transfer`：账户间转账。
- `loan_payment`：还贷，可拆本金、利息、费用。
- `investment_buy` / `investment_sell`：投资买卖。
- `balance_adjustment`：余额校准。

流水状态：

- `pending`：待确认，不进入余额计算。
- `confirmed`：已确认，进入余额计算。
- `void`：作废。

### 3.3 RecurringRuleRecord

周期规则用于工资、房租、房贷、车贷、保险等重复事项。规则不会直接改余额，而是在日期范围内生成 `RecurringOccurrence`，由用户确认后落成流水。

### 3.4 RecurringOccurrence

待确认发生项是首页「待处理」的核心数据：

- 工资 25,000，2026-05-25，待确认。
- 房贷 8,420，2026-05-20，待确认。
- 车贷 3,200，2026-05-15，待确认。

确认后创建一条 `TransactionRecord`，并标记规则的 `lastConfirmedDate`。

## 4. 贷款处理

房贷/车贷继续复用 `src/domain/loanSchedule.ts`。新增单期计划能力后，还款流水可以拆成：

- 银行卡支出：总月供。
- 贷款本金减少：`principalAmount`。
- 利息支出：`interestAmount`。

第一阶段不做提前还款、多段利率和重定价，但数据结构保留 `principalAmount` / `interestAmount` / `feeAmount`，为后续扩展留出口。

## 5. 存储策略

新增 localforage key：

- `accounts`
- `transactions`
- `recurringRules`

保留现有 key：

- `assets`
- `liabilities`

第一阶段只新增底座，不自动迁移旧数据。后续可提供「从当前资产/负债生成账户初始余额」的显式迁移入口。

## 6. 首页交互方向

首页应从纯展示页逐步变成维护入口：

- 净资产摘要。
- 今日/本月待确认。
- 快速记账按钮：支出、收入、转账、还贷。
- 最近流水。
- 行情刷新与估值提醒。

## 7. 验收口径

- 能创建账户、流水、周期规则。
- 周期规则能生成日期范围内的待确认发生项，并跳过已确认日期。
- 待确认发生项确认后写入流水。
- 余额计算只统计 `confirmed` 流水。
- 贷款还款流水能表达本金和利息拆分。
